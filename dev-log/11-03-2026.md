# Dev Log — 11 March 2026

## ~09:00 — Production CORS failure diagnosed

After deploying the WebAuthn changes to the VPS, all API calls from the Vercel frontend were blocked with:
> `Access to fetch blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present`

Root cause: Docker Compose was not auto-loading `.env.backend` (the file was named `.env.backend`, not `.env`). The container started with `CORS_ORIGIN` empty, so only localhost origins were allowed. Fixed by passing `--env-file .env.backend` explicitly to all `docker compose` commands.

**Files affected**: `scripts/vps-up.sh` (noted — not patched, user must use explicit flag)

---

## ~09:30 — Admin bootstrap blocked by legacy `ADMIN_USERNAMES` env var

The `/setup` page was returning "Admin already configured" despite the MongoDB `users` collection being empty. The bootstrap endpoint checks `ADMIN_USERNAMES` env var first — it was still set to `admin` from the old password-based auth system.

Cleared `ADMIN_USERNAMES=` in `.env.backend`, restarted container. Bootstrap worked.

Also cleared `AUTH_USERS` (legacy password seed users) from the env — no longer needed with passkey auth.

Added `WEBAUTHN_RP_ID`, `WEBAUTHN_ORIGIN`, `WEBAUTHN_RP_NAME`, `APP_URL` to `.env.backend`.

**Note**: The `.env.backend` file was accidentally corrupted by a `sed -i` command and had to be recreated from the example template.

## ~10:00 — "Open dashboard" button broken for logged-in org admins

The "Open dashboard" button in `MarketingNav` always navigated to `/app`, but org admins need to go to `/org`. Since `showOrgDashboard` requires `route === '/org'`, clicking the button showed nothing.

**Fix**: Added `profileType` prop to `MarketingNav` and `HomePage`. When `profileType === 'org'` the button navigates to `/org`, otherwise `/app`.

Also moved `profileType` declaration above `marketingPages` in `App.tsx` to avoid "used before declaration" TypeScript error.

**Files changed**:
- `frontend/src/components/MarketingNav.tsx`
- `frontend/src/pages/HomePage.tsx`
- `frontend/src/App.tsx`

## ~11:30 — Backup passkey + emergency recovery

Added two account recovery mechanisms:

### Backup passkey (USB security key)
Platform admin can register additional passkeys from the admin dashboard Security section. Each passkey is stored as a separate credential row in `webauthn_credentials`. Any registered credential can be used to log in.

**Backend**:
- `GET /api/admin/webauthn/backup` — list existing platform credentials (auth required)
- `POST /api/admin/webauthn/backup/begin` — begin registration (auth required, uses logged-in username)
- `POST /api/admin/webauthn/backup/complete` — complete and store new credential (auth required)

**Frontend**:
- `AdminSecurityPage.tsx` (new) — standalone dashboard page at `/app/admin/security`
- `addBackupPasskey` function added to `useAuth.ts`
- "Security" nav item added to `DashboardLayout` sidebar (admin-only)
- `DashboardNav` type extended: `'security'` added
- `useDashboardNav` updated: `security` → `/app/admin/security`

### Emergency recovery page
A hidden recovery page at `/staff-a7f3/recovery` (accessible only from the staff login page via an invisible button). Shows step-by-step MongoDB commands to delete the admin's passkey credentials and user record so the account can be re-bootstrapped from `/setup`.

**Frontend**:
- `AdminRecoveryPage.tsx` (new) — static page with recovery steps
- Hidden `·` button at bottom of `PlatformLoginPage` links to the recovery page
- Route `/staff-a7f3/recovery` wired in `App.tsx`
